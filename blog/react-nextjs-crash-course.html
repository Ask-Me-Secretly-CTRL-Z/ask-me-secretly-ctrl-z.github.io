Setting Up · الإعداد

English: Start by creating a new Next.js project with the App Router. The command is simple:

npx create-next-app@latest my-app --typescript --tailwind --app

English: This sets up TypeScript, Tailwind CSS, and the App Router automatically. The project structure includes app/ for routes, components/ for reusable UI, and public/ for static assets. Each folder inside app/ becomes a route — app/about/page.tsx becomes /about.

English: Once initialized, run npm run dev and you'll see your app at http://localhost:3000. The default page is app/page.tsx. Try editing it — changes reflect instantly thanks to Fast Refresh. Next.js also generates a layout.tsx which wraps every page. This is where you put shared UI like navigation bars and footers.

English: For an optimal developer experience, install additional tools: ESLint for code linting, Prettier for formatting, and the VS Code Tailwind CSS IntelliSense extension. Configure tsconfig.json with path aliases for cleaner imports:

// tsconfig.json
{
  "compilerOptions": {
    "paths": { "@/*": ["./src/*"] }
  }
}
// Now you can import: import { Button } from "@/components/Button"

العربية: ابدأ بإنشاء مشروع Next.js جديد مع App Router. الأمر بسيط: npx create-next-app@latest my-app --typescript --tailwind --app. هذا يهيئ TypeScript و Tailwind CSS و App Router تلقائيًا. هيكل المشروع يشمل app/ للمسارات، components/ للمكونات القابلة لإعادة الاستخدام، و public/ للملفات الثابتة.

العربية: بعد التهيئة، شغّل npm run dev وسترى تطبيقك على http://localhost:3000. الصفحة الافتراضية هي app/page.tsx. جرب تعديلها — التغييرات تظهر فورًا بفضل Fast Refresh. Next.js ينشئ أيضًا layout.tsx الذي يغلف كل صفحة. هنا تضع واجهات المستخدم المشتركة مثل أشرطة التنقل والتذييلات.

العربية: لتجربة مطورين مثلى، ثبّت أدوات إضافية: ESLint لفحص الكود و Prettier للتنسيق وإضافة VS Code Tailwind CSS IntelliSense. هيّئ tsconfig.json بمسميات مسارات للاستيراد الأنظف كما في المثال أعلاه.

Components & Props · المكونات والخصائص

English: React components are functions that return JSX. Props are the way to pass data between components. In React 19, you can use server components by default, and client components by adding "use client" at the top. Here's a simple component:

// app/components/Card.tsx
export default function Card({ title, children }: {
  title: string;
  children: React.ReactNode;
}) {
  return (
    <div className="rounded-xl bg-white/5 p-6 border border-white/10">
      <h3 className="text-lg font-bold mb-2">{title}</h3>
      {children}
    </div>
  );
}

English: This component receives a title prop and children for nested content. The pattern of composing small, focused components is the foundation of React architecture. Each component should do one thing well.

English: Props are read-only — a component should never modify its props. For dynamic behavior, use state. TypeScript helps catch prop-related bugs at compile time. Always type your props with interfaces or type aliases. Consider using React 19's use() hook for reading promises and context within components:

// Using the use() hook in React 19
import { use } from 'react';

function Comments({ commentsPromise }: { commentsPromise: Promise<Comment[]> }) {
  const comments = use(commentsPromise);
  return comments.map(c => <p key={c.id}>{c.text}</p>);
}

العربية: هذا المكون يستقبل خاصية title و children للمحتوى المتداخل. نمط بناء مكونات صغيرة ومركزة هو أساس معمارية رياكت. كل مكون يجب أن يقوم بشيء واحد فقط بإتقان.

العربية: الخصائص للقراءة فقط — لا يجب على المكون أبدًا تعديل خصائصه. للسلوك الديناميكي، استخدم الحالة. TypeScript يساعد في اكتشاف أخطاء الخصائص في وقت التجميع. اكتب دائمًا أنواع خصائصك باستخدام interfaces أو type aliases. استخدم hook use() في React 19 لقراءة الوعود والـ context داخل المكونات كما في المثال أعلاه.

Server Components & Data Fetching · مكونات الخادم وجلب البيانات

English: Next.js 15 introduced stable Server Components. Components are server-rendered by default — they can directly access databases, file systems, and APIs without exposing secrets to the client. Data fetching becomes straightforward:

// app/posts/page.tsx (Server Component - default)
export default async function PostsPage() {
  const res = await fetch('https://api.example.com/posts');
  const posts = await res.json();

  return (
    <div className="grid grid-cols-1 md:grid-cols-3 gap-6">
      {posts.map(post => (
        <PostCard key={post.id} post={post} />
      ))}
    </div>
  );
}

English: This entire component runs on the server. The fetch call, data processing, and initial render all happen server-side. The client only receives the final HTML, making the page load instantly. You can also fetch data incrementally using loading.tsx for streaming and error.tsx for error boundaries.

English: Next.js extends the native fetch API with caching and revalidation options. Use cache: 'no-store' for dynamic data, next: { revalidate: 3600 } for ISR (Incremental Static Regeneration), and default caching for static data. This granular control lets you optimize each data source independently.

// Different caching strategies
// Static - cached indefinitely
const staticData = await fetch('https://api.example.com/settings');
// Revalidated every hour
const posts = await fetch('https://api.example.com/posts', { next: { revalidate: 3600 } });
// Dynamic - never cached
const userData = await fetch('https://api.example.com/me', { cache: 'no-store' });

العربية: هذا المكون بأكمله يعمل على الخادم. عملية جلب البيانات ومعالجتها والتصيير الأولي كلها تتم على الخادم. العميل يستلم فقط HTML النهائي، مما يجعل الصفحة تُحمل فورًا. يمكنك أيضًا جلب البيانات بشكل تدريجي باستخدام loading.tsx للتدفق و error.tsx لإدارة الأخطاء.

العربية: Next.js يوسّع واجهة fetch الأصلية بخيارات التخزين المؤقت وإعادة التحقق. استخدم cache: 'no-store' للبيانات الديناميكية و next: { revalidate: 3600 } للتوليد الثابت المتزايد (ISR) والتخزين المؤقت الافتراضي للبيانات الثابتة. هذا التحكم الدقيق يتيح لك تحسين كل مصدر بيانات بشكل مستقل كما في المثال أعلاه.

Server Actions · إجراءات الخادم

English: Server Actions allow you to run server-side code directly from client components. They're ideal for form handling, database mutations, and any operation that needs server-side execution. Here's an example:

// app/actions.ts
"use server";

export async function createPost(formData: FormData) {
  const title = formData.get('title');
  const content = formData.get('content');

  const post = await db.post.create({
    data: { title, content }
  });

  return { success: true, id: post.id };
}

English: Call this from any client component with useActionState or directly in a form's action prop. React 19's useActionState hook provides pending states and error handling out of the box.

English: Server Actions also support progressive enhancement — forms work even without JavaScript. The action runs on the server, and the page updates automatically. You can also call Server Actions from useEffect or event handlers for more complex interactions. Always validate input on the server side using libraries like Zod or Valibot:

// Server Action with validation
"use server";
import { z } from 'zod';

const schema = z.object({
  email: z.string().email(),
  password: z.string().min(8)
});

export async function signup(prev: any, formData: FormData) {
  const parsed = schema.safeParse({
    email: formData.get('email'),
    password: formData.get('password')
  });
  if (!parsed.success) return { error: parsed.error.flatten() };
  // Proceed with signup...
}

العربية: استدعِ هذا من أي مكون عميل باستخدام useActionState أو مباشرة في خاصية action للنموذج. hook useActionState في React 19 يوفر حالات التحميل ومعالجة الأخطاء مباشرة.

العربية: إجراءات الخادم تدعم أيضًا التحسين التدريجي — النماذج تعمل حتى بدون JavaScript. الإجراء يعمل على الخادم، والصفحة تُحدّث تلقائيًا. يمكنك أيضًا استدعاء إجراءات الخادم من useEffect أو معالجات الأحداث للتفاعلات الأكثر تعقيدًا. تحقق دائمًا من الإدخالات على جانب الخادم باستخدام مكتبات مثل Zod أو Valibot كما في المثال أعلاه.

React 19 New Features · ميزات React 19 الجديدة

English: React 19 introduces several groundbreaking features. The use() hook lets you read promises and context directly in render, simplifying data flow. The new React Compiler automatically memoizes your components, eliminating the need for manual optimizations like useMemo and useCallback in most cases.

English: Server Components, stable since React 19, allow components to run exclusively on the server. This reduces bundle size dramatically. Combined with Server Actions, you can handle form submissions, database writes, and authentication all from within your component tree, without exposing logic to the client.

English: The useActionState hook replaces the experimental useFormState and provides a standard way to handle form state including pending states and errors. useOptimistic lets you update the UI immediately while a server action is processing, creating a snappy user experience. Here's a practical example:

// useOptimistic in action
"use client";
import { useOptimistic } from 'react';

function MessageList({ messages, sendMessage }: {
  messages: Message[];
  sendMessage: (text: string) => Promise<void>;
}) {
  const [optimisticMessages, addOptimistic] = useOptimistic(
    messages,
    (state, newMessage: string) => [...state, { text: newMessage, sending: true }]
  );

  async function handleSubmit(formData: FormData) {
    addOptimistic(formData.get('message'));
    await sendMessage(formData.get('message'));
  }
  // ...render optimistically
}

العربية: React 19 يقدم عدة ميزات ثورية. hook use() يتيح لك قراءة الوعود والـ context مباشرة في التصيير، مما يبسط تدفق البيانات. React Compiler الجديد يقوم تلقائيًا بحفظ المكونات، مما يلغي الحاجة للتحسينات اليدوية مثل useMemo و useCallback في معظم الحالات.

العربية: مكونات الخادم، المستقرة منذ React 19، تسمح للمكونات بالعمل حصريًا على الخادم. هذا يقلل حجم الحزمة بشكل كبير. مع إجراءات الخادم، يمكنك التعامل مع إرسال النماذج وكتابة البيانات في قاعدة البيانات والمصادقة كلها من داخل شجرة المكونات دون كشف المنطق للعميل.

العربية: hook useActionState يحل محل useFormState التجريبي ويوفر طريقة قياسية للتعامل مع حالة النموذج بما في ذلك حالات التحميل والأخطاء. useOptimistic يسمح لك بتحديث واجهة المستخدم فورًا بينما يتم معالجة إجراء الخادم، مما يخلق تجربة مستخدم سريعة كما في المثال أعلاه.

State Management · إدارة الحالة

English: For global state, React 19 includes use() hook for reading context and promises directly. Combine with Zustand or Jotai for complex state needs. Server Components eliminate much of the need for client-side state management — data is fetched on the server and passed down.

English: When you do need client state, keep it minimal. Start with React's built-in useState and useReducer. For global state that doesn't change often, use React Context. For frequently updated global state, choose Zustand — it's minimal and TypeScript-friendly:

// Zustand store example
import { create } from 'zustand';

interface CartStore {
  items: Item[];
  addItem: (item: Item) => void;
  total: () => number;
}

export const useCartStore = create<CartStore>((set, get) => ({
  items: [],
  addItem: (item) => set((state) => ({ items: [...state.items, item] })),
  total: () => get().items.reduce((sum, i) => sum + i.price, 0),
}));

English: Understand the difference between server state and client state. Server state (user data, posts, comments) should be fetched and cached with tools like React Query or SWR. Client state (theme, modals, form inputs) stays in React state. Mixing the two leads to bugs and poor performance.

العربية: للحالات المعقدة، استخدم Zustand — بسيط ومتوافق مع TypeScript ويعمل جيدًا مع مكونات الخادم والعميل. تجنب الإفراط في هندسة إدارة الحالة؛ معظم التطبيقات تحتاج فقط إلى حالة محلية وبيانات من الخادم.

العربية: افهم الفرق بين حالة الخادم وحالة العميل. حالة الخادم (بيانات المستخدم، المقالات، التعليقات) يجب جلبها وتخزينها مؤقتًا باستخدام أدوات مثل React Query أو SWR. حالة العميل (السمة، النوافذ المنبثقة، إدخالات النماذج) تبقى في حالة رياكت. خلط الاثنين يؤدي إلى أخطاء وأداء ضعيف.

العربية: مثال متجر Zustand أعلاه يوضح كيفية إنشاء متجر بسيط لإدارة سلة التسوق مع دعم TypeScript الكامل. هذا النمط مثالي للحالات التي تحتاجها عبر عدة مكونات مثل عربة التسوق أو تفضيلات المستخدم.

Routing & Layouts · التوجيه والتخطيطات

English: Next.js App Router uses a file-system based routing. Create parallel routes, intercepting routes, and dynamic routes with simple folder structures. Layouts persist across navigations and can be nested:

app/
  layout.tsx          // Root layout (applies everywhere)
  page.tsx            // Home page (/)
  about/
    page.tsx          // About page (/about)
  blog/
    layout.tsx        // Blog layout (applies to all blog pages)
    page.tsx          // Blog index (/blog)
    [slug]/
      page.tsx        // Dynamic route (/blog/my-post)

English: This structure provides automatic code splitting, prefetching, and caching. Use generateStaticParams for static generation of dynamic routes. For complex navigation patterns, use parallel routes (two pages in one view) and intercepting routes (modals that work with URL sharing).

English: Layouts are unique in Next.js — they don't unmount when navigating between pages. This means you can put sidebars, navigation, and audio players in a layout and they'll persist seamlessly. Nested layouts inherit from parent layouts, allowing for section-specific navigation:

// app/dashboard/layout.tsx - Dashboard layout
export default function DashboardLayout({ children, analytics }: {
  children: React.ReactNode;
  analytics: React.ReactNode;
}) {
  return (
    <div className="flex">
      <DashboardSidebar />
      <main>{children}</main>
      {analytics}
    </div>
  );
}

العربية: هذا الهيكل يوفر تقسيم الكود تلقائيًا والجلب المسبق والتخزين المؤقت. استخدم generateStaticParams للتوليد الثابت للمسارات الديناميكية. للأنماط المعقدة، استخدم المسارات المتوازية (صفحتان في عرض واحد) والمسارات المعترضة (نوافذ منبثقة تعمل مع مشاركة URL).

العربية: التخطيطات فريدة في Next.js — لا تُفصل عند التنقل بين الصفحات. هذا يعني أنه يمكنك وضع الأشرطة الجانبية والتنقل ومشغلات الصوت في تخطيط وستبقى بشكل سلس. التخطيطات المتداخلة ترث من التخطيطات الأم، مما يتيح تنقلًا خاصًا بكل قسم.

Performance Optimization · تحسين الأداء

English: Performance is critical for user retention and SEO. Next.js provides built-in performance features: image optimization with next/image, font loading with next/font, and script loading with next/script. Use Partial Prerendering (PPR) for hybrid static + dynamic pages.

English: For client components, apply these patterns: use React.memo sparingly (the React Compiler handles this automatically in React 19), implement code splitting with next/dynamic, and virtualize long lists with libraries like @tanstack/virtual. Always monitor Core Web Vitals using tools like Lighthouse and Web Vitals:

// Dynamic import for code splitting
import dynamic from 'next/dynamic';

const HeavyComponent = dynamic(() => import('@/components/HeavyChart'), {
  loading: () => <p>Loading chart...</p>,
  ssr: false, // Disable SSR if component uses browser APIs
});

English: Image optimization alone can cut page weight by 50% or more. Always provide width, height, and priority for above-the-fold images. Use WebP format and responsive srcSet. Next.js automatically serves modern formats and resizes images on demand.

العربية: الأداء ضروري للاحتفاظ بالمستخدمين وتحسين محركات البحث. Next.js يوفر ميزات أداء مدمجة: تحسين الصور بـ next/image وتحميل الخطوط بـ next/font وتحميل النصوص البرمجية بـ next/script. استخدم التصيير المسبق الجزئي (PPR) للصفحات الهجينة.

العربية: لمكونات العميل، طبق هذه الأنماط: استخدم React.memo بشكل محدود (React Compiler يتولى هذا تلقائيًا في React 19)، طبق تقسيم الكود بـ next/dynamic، وقم بتفعيل القوائم الطويلة بمكتبات مثل @tanstack/virtual. راقب دائمًا Core Web Vitals باستخدام أدوات مثل Lighthouse كما في المثال أعلاه.

Testing React Applications · اختبار تطبيقات رياكت

English: Testing ensures your application works correctly and prevents regressions. The standard stack is Vitest (or Jest) + React Testing Library. Vitest is faster and integrates seamlessly with Vite-based Next.js projects. React Testing Library encourages testing behavior, not implementation details.

English: Write unit tests for individual components, focusing on user interactions and rendered output. Use screen queries to find elements by role, text, or test ID — not by CSS class or tag name. Test accessibility roles as part of your tests:

// Component test with React Testing Library
import { render, screen, fireEvent } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import Counter from './Counter';

test('increments count when button clicked', async () => {
  render(<Counter />);
  
  const button = screen.getByRole('button', { name: /increment/i });
  await userEvent.click(button);
  
  expect(screen.getByText('Count: 1')).toBeInTheDocument();
});

English: For server components, test data fetching and rendering separately. Use vi.mock to mock fetch calls. Write integration tests for critical user flows like signup, checkout, or search. Aim for the testing trophy — a few E2E tests, several integration tests, and many unit tests.

العربية: الاختبار يضمن أن تطبيقك يعمل بشكل صحيح ويمنع الانتكاسات. الحزمة القياسية هي Vitest (أو Jest) مع React Testing Library. Vitest أسرع ويتكامل بسلاسة مع مشاريع Next.js القائمة على Vite. React Testing Library تشجع على اختبار السلوك وليس تفاصيل التنفيذ.

العربية: اكتب اختبارات وحدة للمكونات الفردية، مركزًا على تفاعلات المستخدم والمخرجات المصيّرة. استخدم استعلامات screen للعثور على العناصر بواسطة الدور أو النص أو معرّف الاختبار — ليس بواسطة class CSS أو اسم الوسم. اختبر أدوار الوصول كجزء من اختباراتك كما في المثال أعلاه.

Deployment Strategies · استراتيجيات النشر

English: Next.js offers multiple deployment options. Vercel is the most seamless — one click from GitHub and you get preview deployments, automatic SSL, and edge functions. For self-hosted deployments, use Docker with a Node.js server, or export to static HTML with next export for CDN hosting.

English: Configure environment variables properly: use .env.local for local development, .env.production for production, and never commit secrets. Set up CI/CD pipelines with GitHub Actions, GitLab CI, or Jenkins that run linting, tests, and type-checking before deployment:

# .github/workflows/deploy.yml
name: Deploy to Production
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm run lint
      - run: npm run build
      - uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}

English: Monitor your deployed application with tools like Sentry for error tracking, Vercel Analytics for performance, and Logtail for server logs. Set up alerts for critical errors and performance degradation. A well-deployed application includes monitoring, logging, and a rollback strategy.

العربية: Next.js يقدم خيارات نشر متعددة. Vercel هو الأكثر سلاسة — نقرة واحدة من GitHub وستحصل على نشر تجريبي و SSL تلقائي و edge functions. للنشر الذاتي، استخدم Docker مع خادم Node.js، أو صدر إلى HTML ثابت بـ next export للاستضافة على CDN.

العربية: هيّئ متغيرات البيئة بشكل صحيح: استخدم .env.local للتطوير المحلي و .env.production للإنتاج ولا ترسل الأسرار أبدًا. أعد خطوط CI/CD مع GitHub Actions أو GitLab CI أو Jenkins التي تشغل فحص الكود والاختبارات وفحص الأنواع قبل النشر كما في المثال أعلاه.

Real-World Project Structure · هيكل مشروع حقيقي

English: A production Next.js project needs more than just pages. Organize your codebase with a clear structure that scales. Here's a recommended structure for a medium-sized application:

src/
  app/                    # Routes
    (auth)/               # Route group (no layout inheritance)
      login/page.tsx
    dashboard/
      layout.tsx
      page.tsx
  components/
    ui/                   # Reusable UI primitives
    forms/                # Form components
    layout/               # Navigation, sidebar, footer
  lib/
    db.ts                 # Database client
    auth.ts               # Auth utilities
    api-client.ts         # External API client
  hooks/
    use-auth.ts
    use-debounce.ts
  types/
    index.ts
  utils/
    cn.ts                 # Classname utility (clsx + twMerge)
    format.ts             # Date, currency formatting

English: Use route groups (folders in parentheses) to organize routes without affecting the URL path. Keep components close to where they're used — put shared components in components/ui/ and page-specific components next to the page. This structure scales from a small blog to a large SaaS application.

العربية: مشروع Next.js إنتاجي يحتاج أكثر من مجرد صفحات. نظّم قاعدة الكود بهيكل واضح يتوسع. استخدم مجموعات المسارات (مجلدات بين قوسين) لتنظيم المسارات دون التأثير على مسار URL. ابقِ المكونات قريبة من مكان استخدامها — ضع المكونات المشتركة في components/ui/ والمكونات الخاصة بالصفحة بجوار الصفحة. هذا الهيكل يتوسع من مدونة صغيرة إلى تطبيق SaaS كبير.
